# 熔断器问题复盘与改进建议

## 文档目的

本文总结本次熔断器实现与测试过程中遇到的关键问题，覆盖：

- 设计层问题（语义边界、时间窗口模型）
- 实现层问题（并发安全、统计正确性、可用性）
- 对应修复方案与推荐优化路径

## 验证结论（当前状态）

- 已通过：`internal/cli/breaker` 包的单测与 race 检测
- 执行命令：`CGO_ENABLED=1 go test -race ./internal/cli/breaker -v`

## 设计层问题

### 1. 时间窗口语义不清晰

问题表现：
- 对滑动窗口是“覆盖最旧桶”还是“满了就丢当前桶”缺少明确定义。
- `advance(now)` 在大时间跨度下应如何补齐空桶、何时清窗，边界不够明确。

影响：
- 不同实现者可能得出不同统计行为。
- 熔断判定与预期不一致，出现偶发偏差。

推荐方案：
- 明确采用“尾部追加 + 淘汰最旧（tail-append / oldest-evict）”语义。
- 规定 `advance(now)` 的职责：
  1. 时间推进
  2. 统计对齐
  3. 过期清理
  4. 作为 Allowed 与 Record 的统一时间基准

### 2. Allowed 与 Record 的职责边界模糊

问题表现：
- 状态迁移逻辑容易混杂在两个入口，可能重复判断或遗漏。

影响：
- 代码可维护性下降，且更容易在后续重构中引入回归。

推荐方案：
- Allowed 负责“放行前决策”相关状态切换：
  - Closed -> Open（基于时间窗失败率）
  - Open -> HalfOpen（开路时间到期）
  - HalfOpen 超时 -> Open
- Record 负责“结果回写”相关状态切换：
  - HalfOpen 失败 -> Open
  - HalfOpen 连续成功达到阈值 -> Closed

## 实现层问题

### 1. P0：首次 Allowed 的空指针崩溃（已修复）

问题表现：
- 对新 key 调用 Allowed 时，map 中不存在子熔断器；创建后未绑定到局部变量，后续直接调用导致 nil 解引用。

根因：
- 查找/创建与使用之间变量未保持一致。

修复方案：
- 在同一临界区内完成“查找或创建并返回实例”。
- 当前已修复。

### 2. P1：TOCTOU 竞态导致实例覆盖与统计丢失（已修复）

问题表现：
- 旧流程为先读锁检查，再写锁创建；并发时两个协程可能同时判断不存在并各自创建实例，后写覆盖先写。

根因：
- TOCTOU（Time Of Check To Time Of Use）：检查结果在使用前已失效。

修复方案：
- 使用写锁包住“检查 + 创建 + 使用”的完整流程，避免检查与使用分离。
- 当前已修复。

### 3. 统计可见性延迟（已修复）

问题表现：
- Record 后立刻 Allowed，在某些实现下新数据只在 current 桶，未参与 Summary，导致熔断触发滞后。

修复方案：
- 让 Summary 同时汇总历史窗口与 current 桶，保证最新请求立即可见。
- 当前已修复。

### 4. P1：入队失败可能静默丢桶（待优化）

问题表现：
- 窗口容器入队返回值未检查，队列满时可能丢失推进事件。

影响：
- 统计窗口可能偏离真实时间线，造成判定失真。

推荐方案：
- 将窗口容器改为显式 oldest-evict 行为。
- 或在入队失败时先出队再入队，并增加监控计数。

### 5. P1：advance 大跨度推进持锁循环（待优化）

问题表现：
- 时间跨度大时，多次循环推进桶，持锁时间随跨度增长。

影响：
- 同 key 请求尾延迟上升。

推荐方案：
- 增加批量推进 API。
- 当跨度 >= 容量时直接 reset，避免线性循环。

### 6. P2：Summary 热路径 O(window_size)（待优化）

问题表现：
- 每次判定都扫描窗口桶。

影响：
- 高频场景 CPU 开销明显，且放大锁竞争。

推荐方案：
- 引入 rolling summary（增量加减）降为 O(1)。

## 本次测试与质量保障

- 状态机测试：Closed / Open / HalfOpen 三大状态迁移
- 并发测试：
  - HalfOpen 并发放行不超配额
  - Allowed 与 Record 并发不死锁
- Race 检测：已通过

## 推荐落地顺序

1. 保持当前修复（P0/P1 已解决部分）并加回归测试防回退。
2. 优先处理窗口入队语义（避免静默丢桶）。
3. 优化 advance 批量推进，降低持锁时长。
4. 视流量规模引入 rolling summary，解决热路径复杂度问题。

## 附：TOCTOU 简述

TOCTOU 是并发编程中的经典问题，指“检查时成立的条件，在真正使用时可能已经失效”。

在本项目中的典型例子：

- 先判断 map 不存在 key（Check）
- 释放读锁
- 再去创建/使用该 key（Use）

如果中间有其他协程修改了 map，就会导致覆盖、错用旧引用或数据丢失。最佳实践是把 Check 与 Use 放在同一原子临界区内。
