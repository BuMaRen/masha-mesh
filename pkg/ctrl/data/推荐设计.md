# EndpointSlice 缓存推荐设计

## 一、设计目标

本设计目标是把“缓存”从 `Logic` 和“发布流程”中解耦，形成可独立测试、可并发安全复用的组件。

目标包括：

1. 缓存职责单一：只负责状态维护与查询。
2. 事件处理与发布可插拔：通过独立 Reconciler 负责发布策略。
3. 支持后续扩展：去重策略、增量合并、重试、限流。

## 二、推荐分层

建议拆成三层：

1. **Cache 层（领域状态）**
2. **Reconciler 层（事件编排）**
3. **Logic 层（基础设施装配）**

### 1. Cache 层

仅做三件事：

1. 接收变更并更新内存状态。
2. 维护 service 粒度的 EndpointSlice 聚合索引。
3. 对外提供 `GetMerged(service)` 查询。

Cache 层不做：

1. 发布消息。
2. 依赖 gRPC 或 informer 具体实现。

### 2. Reconciler 层

职责：

1. 接收 informer 事件。
2. 调用 cache 的 Apply 接口。
3. 根据返回结果判断事件类型并调用 `Distributer.Publish`。

### 3. Logic 层

职责：

1. 组装 `cache + reconciler + distributer`。
2. 注册 informer 回调到 reconciler。
3. 负责进程生命周期（启动、退出、优雅停机）。

## 三、推荐接口

## 1. Cache 接口

```go
type EndpointSliceCache interface {
	 ApplyAdd(es *discoveryv1.EndpointSlice) (changed bool, svc string)
	 ApplyUpdate(oldEs, newEs *discoveryv1.EndpointSlice) (changed bool, svc string)
	 ApplyDelete(es *discoveryv1.EndpointSlice) (changed bool, svc string, deletedAll bool)
	 GetMerged(svc string) (*discoveryv1.EndpointSlice, bool)
}
```

说明：

1. `changed=false` 时，Reconciler 可跳过 Publish。
2. `deletedAll=true` 表示 service 下已无 EndpointSlice，可发布 `DELETED`。

## 2. Reconciler 接口（可选）

```go
type EndpointSliceReconciler interface {
	 OnAdded(obj any)
	 OnUpdate(oldObj, newObj any)
	 OnDeleted(obj any)
}
```

## 四、数据结构建议

建议在缓存内部维护：

1. `serviceMap map[string]*ServiceSlices`
2. `ServiceSlices.slices map[string]*discoveryv1.EndpointSlice`
3. 可选：`mergedCache *discoveryv1.EndpointSlice` + `dirty bool`

其中 `ServiceSlices` 的含义是“某个 service 的所有源 EndpointSlice 状态”。

## 五、并发模型建议

至少采用 `sync.RWMutex` 保护缓存：

1. 写操作（ApplyAdd/Update/Delete）加写锁。
2. 读操作（GetMerged）加读锁。

如果后续并发提升，可升级为：

1. 按 service 分片锁。
2. 或按 service key 串行 worker。

## 六、事件发布策略建议

Reconciler 发布建议：

1. Add 且 changed：发布 `ADDED`（首次）或 `MODIFIED`（非首次，可按业务约定）。
2. Update 且 changed：发布 `MODIFIED`。
3. Delete 且 changed：
	1. `deletedAll=true` -> 发布 `DELETED`；
	2. 否则发布 `MODIFIED`。

同时建议在 Reconciler 层做以下控制：

1. 类型断言失败日志。
2. tombstone 兼容处理。
3. 发布前可选比对 merged 结果是否真实变化，避免无效推送。

## 七、推荐目录形态

可在 `pkg/ctrl/cache/` 下组织：

1. `endpoint_cache.go`：缓存实现。
2. `service_slices.go`：service 聚合结构与 merge。
3. `reconciler.go`：事件到发布的编排。
4. `types.go`：接口定义。
5. `*_test.go`：缓存单测与 reconciler 单测。

## 八、迁移步骤（低风险）

建议按以下顺序迁移，保证行为可回归：

1. **抽离缓存实现**
	1. 从现有 `CoreData` 中抽出纯状态逻辑到新 cache。
	2. 保留原 merge 规则与版本检查。
2. **引入 Reconciler**
	1. 将 Publish 从 `CoreData` 移到 Reconciler。
	2. Reconciler 依赖 `cache + distributer`。
3. **改造 Logic 组装**
	1. `NewLogic` 里构建 cache/reconciler。
	2. informer 回调由 `l.core.*` 改成 `reconciler.*`。
4. **回归测试**
	1. 迁移原有 storage 测试到 cache/reconciler。
	2. 保证事件序列输出行为不变。

## 九、设计收益

完成后可获得：

1. 组件边界清晰，缓存不再“附属于 logic”。
2. 代码可测试性提升，定位问题更快。
3. 并发安全和扩展能力更强。
4. 后续做增量 merge、事件抑制、重试更自然。

## 十、结论

推荐采用“**缓存实例化 + 事件编排解耦**”方案：

1. 缓存作为独立实例存在于控制器生命周期内。
2. Logic 只负责装配，不承载领域细节。
3. Reconciler 处理 informer 事件和发布策略。

这比当前 `CoreData` 一体化实现更符合 Go 在控制器场景下的工程实践。
