# masha-mesh 岗位弱项补齐执行手册（按短期收益排序）

## 1. 使用方式

目标：不靠背问答，通过补功能来提升岗位匹配度与面试通过率。

执行原则：
- 每个任务都按 问题定义 -> 实现 -> 验证 -> 复盘 走完闭环。
- 每个任务都必须有可展示证据：代码变更、测试结果、文档结论。
- 先做高收益低风险项，再做中长期加分项。

建议节奏：每周完成 1 个 P0/P1 任务闭环。

---

## 2. 优先级总览（高 -> 低）

P0（最高）：AI 准入策略从“注入”升级为“治理入口”
- 直接对应 JD 的 Kubernetes、Operator、AI 基础设施能力。
- 现有项目基础最好，改动收益最大。
- 预计 3 到 5 天可形成可演示成果。

P1：可用性工程补齐（探针、优雅停机、失败策略）
- 直接对应 JD 的生产稳定性与故障处理能力。
- 面试问到高频，且可在本仓库快速补完。
- 预计 2 到 4 天。

P2：CI/CD 工程化落地（从脚本到流水线）
- 直接对应 JD 的自动化交付能力。
- 可快速将现有 deploy 脚本转成流水线能力。
- 预计 2 到 3 天。

P3：Istio 实战补齐（最小可行集成）
- 直接补齐你当前最明显短板。
- 对面试说服力很强，但投入略高。
- 预计 4 到 7 天。

P4：云平台实战证据（最小生产形态）
- 解决云平台经验不足问题。
- 时间与环境成本最高，放在后面。
- 预计 5 到 10 天。

## 3. P0 任务：AI 准入治理（最高优先）

### 3.1 目标

将当前 webhook 从“只注入 sidecar”升级为“AI workload 准入策略入口”。

### 3.2 关联文件

- internal/ctrl/webhook/aggregation.go
- internal/ctrl/webhook/options.go
- internal/ctrl/options.go
- build/ctrl/deployment.yml
- build/ctrl/webhook.yml
- internal/ctrl/webhook/aggregation_test.go（新增）

### 3.3 详细步骤

步骤 1：定义 AI workload 标识规则
- 规则：当 Pod 带标签 masha.io/ai-enabled=true 时，进入 AI 准入检查。
- 在 aggregation.go 的 mutate 处理路径中增加分支。

步骤 2：增加最小硬策略（先 2 条）
- 策略 A：所有容器必须声明 requests 与 limits（cpu、memory）。
- 策略 B：必须带业务归属标签，默认 masha.io/owner。
- 对违规请求返回 deny，并给出明确 message。

步骤 3：增加策略开关与参数化
- 在 options.go 增加参数：
  - --ai-policy-enforce（默认 false）
  - --ai-required-owner-label（默认 masha.io/owner）
- 在 ctrl options 聚合中透传到 webhook options。
- 在 deployment.yml 中加启动参数，支持灰度启用。

步骤 4：完善可观测日志
- 在准入日志中统一字段：namespace、name、ai_enabled、policy_result、reject_reason。
- 目标：定位问题时不靠模糊日志文本。

步骤 5：补单测
- 新建 aggregation_test.go，覆盖至少 4 个场景：
  - 非 AI workload -> allow
  - AI workload 且资源完整、owner 存在 -> allow
  - AI workload 缺失资源 -> deny
  - AI workload 缺失 owner -> deny

步骤 6：验证与演示
- 构造 3 个示例 Pod YAML（合法、缺资源、缺 owner）。
- 分别执行 kubectl apply，观察 admission 响应是否符合预期。

### 3.4 验收标准

- 策略可以通过启动参数开关控制。
- 违规 Pod 明确拒绝并返回可读 message。
- 单测通过并覆盖关键分支。
- 可提供一次从提交到验证的演示记录。

### 3.5 面试价值

你可以讲出完整链路：策略定义、灰度开关、失败反馈、测试闭环。

## 4. P1 任务：可用性工程（稳定性与故障能力）

### 4.1 目标

补齐控制面在生产场景最常被问的能力：探针、优雅停机、失败策略。

### 4.2 关联文件

- internal/ctrl/logic.go
- internal/ctrl/webhook/server.go
- pkg/metrics/server.go
- build/ctrl/deployment.yml
- build/ctrl/webhook.yml
- docs/webhook-consistency-design.md（新增）

### 4.3 详细步骤

步骤 1：明确 webhook failurePolicy 与 timeout
- 当前 webhook.yml 已配置 failurePolicy 与 timeoutSeconds。
- 补设计文档解释为何选择当前策略，何时切换策略。
- 给出风险说明：可用性优先还是一致性优先。

步骤 2：新增 healthz 与 readyz
- 在 webhook server 增加健康接口。
- readyz 需依赖关键依赖就绪状态，不是简单返回 200。
- 在 deployment.yml 配置 livenessProbe 与 readinessProbe。

步骤 3：补 preStop 与 shutdown 顺序
- 使用 preStop 先摘流量，再执行进程退出。
- 现有代码已有 context shutdown 逻辑，补超时与日志细化。

步骤 4：补失败场景演练
- 场景 A：证书文件错误。
- 场景 B：webhook 响应超时。
- 场景 C：滚动更新过程中的请求稳定性。
- 每个场景都记录现象、根因、修复动作。

### 4.4 验收标准

- 探针生效，滚动发布时没有明显请求中断。
- shutdown 路径可观察、可复现。
- 有故障演练记录文档。

### 4.5 面试价值

你可以回答“生产故障怎么定位和防止复发”，而不是只讲功能。

## 5. P2 任务：CI/CD 流水线化（短平快高收益）

### 5.1 目标

把当前 deploy.sh 与 deploy-ctrl.sh 的能力迁移到标准流水线，形成工程化交付证据。

### 5.2 关联文件

- .github/workflows/ci.yml（新增）
- .github/workflows/cd.yml（新增）
- deploy.sh
- deploy-ctrl.sh
- README.md

### 5.3 详细步骤

步骤 1：建立最小 CI
- 触发条件：pull request、push。
- 流程：go mod download -> go test -> go vet -> go build。
- 输出：构建与测试结果可追溯。

步骤 2：建立最小 CD
- 触发条件：tag 或手动触发。
- 流程：构建镜像 -> 推送镜像 -> 渲染部署版本 -> kubectl apply。
- 复用现有 deploy 脚本作为流水线步骤，减少重写成本。

步骤 3：加入质量门禁
- 主分支要求 CI 通过才能合并。
- 失败时必须有可读日志和失败阶段标记。

步骤 4：更新 README
- 写明 CI/CD 触发方式、输入参数、回滚路径。

### 5.4 验收标准

- 提交 PR 可自动跑 CI。
- 发布 tag 可自动触发部署流程。
- 发布失败时可定位到具体步骤。

### 5.5 面试价值

可直接展示“从代码到部署”的工程化能力，不再只依赖本地脚本。

## 7. P4 任务：云平台最小生产实践

### 7.1 目标

### 7.2 建议路径

### 7.3 详细步骤

- 按现有 build 配置部署 mesh-ctrl。

步骤 3：接入托管可观测
- 将 metrics 接入托管 Prometheus。

步骤 4：做一次故障演练
- 模拟 webhook 不可用或证书过期。
- 输出恢复过程、耗时、改进项。

### 7.4 验收标准

- 有云上部署记录、监控截图、故障演练记录。

## 18. 每日完成判定（防止拖延）
