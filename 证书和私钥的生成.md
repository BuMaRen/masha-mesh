# Kubernetes 场景下证书与私钥生成（含 san.conf / tls.csr）

本文目标：生成可用于 Kubernetes 的 TLS 证书和私钥，并导入集群作为 `kubernetes.io/tls` Secret。

> 结论先看：
> - 本地学习可用「一步自签」命令。
> - 对接 Kubernetes（尤其 Ingress / 域名访问）建议走「`key + csr + san.conf + 签发`」流程。

---

## 1) 一步自签（你现在在用）

命令：

```bash
openssl req -x509 -newkey rsa:2048 -keyout server.key -out server.crt -days 365 -nodes
```

参数解释：
- `-x509`：直接产出证书（自签），不经过 CA。
- `-newkey rsa:2048`：生成新密钥对。
- `-keyout server.key`：私钥输出路径。
- `-out server.crt`：证书输出路径。
- `-days 365`：证书有效期。
- `-nodes`：私钥不加密（便于服务直接加载）。

适用：本地调试、快速验证 TLS。

限制：默认不被浏览器/系统信任，常需要 `curl -k`。

---

## 2) Kubernetes 推荐流程（`san.conf + tls.csr`）

这个流程是你提到的「需要 `san.conf`、`tls.csr`」方式，适合对接 Ingress 和正式环境。

### 步骤 0：明确访问域名（最关键）

你要先确定业务访问地址，例如：

- `api.example.com`（公网域名）
- `mesh.example.com`
- 仅内网可加：`service.namespace.svc.cluster.local`

这些值要写进 SAN（Subject Alternative Name），否则会出现域名不匹配。

### 步骤 1：准备 `san.conf`

创建文件 `san.conf`：

```conf
[ req ]
default_bits       = 2048
prompt             = no
default_md         = sha256
req_extensions     = req_ext
distinguished_name = dn

[ dn ]
C  = CN
ST = Shanghai
L  = Shanghai
O  = BuMaRen
OU = Dev
CN = api.example.com

[ req_ext ]
subjectAltName = @alt_names

[ alt_names ]
DNS.1 = api.example.com
DNS.2 = mesh.example.com
# 如果确实要按 IP 访问，再补 IP
# IP.1 = 127.0.0.1
```

这一步做什么：
- 声明证书主体信息（`CN`）。
- 声明 SAN 列表（现代客户端主要看 SAN）。

### 步骤 2：生成私钥 `tls.key`

```bash
openssl genrsa -out tls.key 2048
```

这一步做什么：
- 生成服务端私钥，只保存在服务端。
- 后续所有签名请求都基于这个私钥。

### 步骤 3：生成 CSR `tls.csr`

```bash
openssl req -new -key tls.key -out tls.csr -config san.conf
```

这一步做什么：
- 生成证书签名请求（CSR）。
- CSR 内包含公钥 + 主体信息 + SAN 扩展请求。
- 你把 `tls.csr` 提交给 CA（公有 CA、企业 CA、或 cert-manager 的签发器）。

### 步骤 4：签发证书（两种）

#### A. 临时自签（联调用）

```bash
openssl x509 -req -in tls.csr -signkey tls.key -out tls.crt -days 365 -extensions req_ext -extfile san.conf
```

这一步做什么：
- 用自己的私钥给 CSR 签发一张自签证书。
- 产物是 `tls.crt` + `tls.key`，可直接导入 Kubernetes。

#### B. 正式 CA 签发（生产推荐）

把 `tls.csr` 提交给 CA，拿回：
- `tls.crt`（站点证书）
- 可能还会有中间证书链（需要合并）

如果 CA 给了链文件（例如 `intermediate.crt`），通常要合并：

```bash
cat tls.crt intermediate.crt > tls-fullchain.crt
```

后续在 K8s 中优先使用 fullchain（按你的 Ingress 控制器要求）。

### 步骤 5：校验证书内容

```bash
openssl x509 -in tls.crt -noout -text
```

重点检查：
- `Subject` 是否正确。
- `X509v3 Subject Alternative Name` 是否包含目标域名。
- `Not Before / Not After` 有效期。

检查私钥与证书是否匹配：

```bash
openssl x509 -noout -modulus -in tls.crt | openssl md5
openssl rsa  -noout -modulus -in tls.key | openssl md5
```

两行哈希值相同才说明配对正确。

## 3) 导入 Kubernetes

### 方式 A：命令行创建 TLS Secret（最常用）

```bash
kubectl -n <namespace> create secret tls masha-mesh-tls \
	--cert=tls.crt \
	--key=tls.key
```

这一步做什么：
- 生成类型为 `kubernetes.io/tls` 的 Secret。
- 键名固定为 `tls.crt` 和 `tls.key`。

### 方式 B：YAML（适合 GitOps）

```yaml
apiVersion: v1
kind: Secret
metadata:
	name: masha-mesh-tls
	namespace: default
type: kubernetes.io/tls
data:
	tls.crt: <base64-encoded-cert>
	tls.key: <base64-encoded-key>
```

---

## 4) 在 Ingress 中引用

```yaml
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
	name: masha-mesh
	namespace: default
spec:
	tls:
		- hosts:
				- api.example.com
			secretName: masha-mesh-tls
	rules:
		- host: api.example.com
			http:
				paths:
					- path: /
						pathType: Prefix
						backend:
							service:
								name: masha-mesh
								port:
									number: 9987
```

关键点：
- `spec.tls.hosts` 必须与证书 SAN 匹配。
- `secretName` 必须是同 namespace 下的 TLS Secret。

---

## 5) 常见错误与排查

- 访问报域名不匹配：证书 SAN 没包含当前 host。
- Secret 创建失败：`tls.crt`/`tls.key` 文件路径错或格式错。
- Ingress 不生效：`secretName` 在错误 namespace。
- 仍需 `-k`：证书是自签，客户端不信任（预期现象）。

---

## 6) 你当前项目如何落地（最短路径）

1. 先按「步骤 1~4A」生成 `tls.crt`、`tls.key`（自签，含 SAN）。
2. 用 `kubectl create secret tls` 导入集群。
3. 在 Ingress `tls.secretName` 引用该 Secret。
4. 域名联通后改为 CA 正式签发证书（步骤 4B）。

这样可以先跑通链路，再平滑切到生产证书。
