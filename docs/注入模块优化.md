# 注入模块优化 TodoList（按代码现状重排）

> 更新时间：2026-06-09
> 范围：仅注入链路（webhook + container reconciler）
> 运行前提：当前按单 namespace 部署和使用（webhook 通过 namespaceSelector 定向到该命名空间）。

## 1. 优化目标（细化版）

### P0 目标（必须完成）
- [ ] 注入规则说清楚且可验证：
  - [ ] pod 上哪个 label 表示要注入。
  - [ ] label 的值表示要注入哪个容器模板名。
  - [ ] webhook 和 controller 用同一套规则，不会一个按 key 判断、另一个按 value 判断。
  - [ ] 这些规则有单测，改代码时能第一时间发现行为变化。
- [ ] 模板查找键先按当前部署前提落地：
  - [ ] 当前单 namespace 场景下，先使用 containerName 作为缓存主键。
  - [ ] 在文档里写明这是有前提的设计，不是通用设计。
  - [ ] 预留多 namespace 扩展方案（需要时再升级为 namespace/containerName）。
- [ ] Admission 入口检查补全：只处理 Pod 的 CREATE 请求，其他请求直接放行或拒绝（策略要写清楚）。
- [ ] 收敛范围收窄：controller 更新 workload 时，不只看“有这个 label key”，还要看 label value 是否等于目标容器名。
- [ ] 删除路径加保险：删除注入容器后，如果 pod 业务容器会变成空，禁止更新并记录原因。
- [ ] 注入链路有基本监控：至少能看到注入成功/失败、cache miss、更新冲突次数、处理耗时。
- [ ] 关键路径有回归测试：mutate handler、Add/Update/Delete 收敛、冲突重试、cache miss、删除安全护栏。

### P1 目标（建议完成）
- [ ] rollout 流程拆成两段：先算出要改哪些对象，再执行更新，便于定位失败点。
- [ ] workload 更新有边界重试：冲突时重试，但要有最大次数，超过后给出明确失败原因。
- [ ] webhook 失败策略参数化：先明确支持 fail-close，后续如要加 fail-open 再扩展。

## 3. 基线核对（基于当前代码）

### 已具备
- [x] 启动层共享 cache 注入 webhook + reconciler（internal/ctrl/logic.go）。
- [x] webhook 已具备最小注入流程：label 判定 + 同名容器判重 + cache 命中后 patch（internal/ctrl/webhook/aggregation.go）。
- [x] controller 已处理 Add/Update/Delete，并更新 Deployment/StatefulSet（internal/ctrl/reconciler/container_reconciler.go）。
- [x] 已有少量单测覆盖纯函数逻辑（internal/ctrl/webhook/aggregation_test.go、internal/ctrl/reconciler/container_test.go）。

### 关键缺口（按优先级）
- [ ] 注入规则还没形成“明确口径 + 测试用例”的组合，后续维护容易出现理解偏差。
- [ ] Admission 请求仅做 JSON/Pod 解析，未校验 Kind/Operation/SubResource（internal/ctrl/webhook/aggregation.go）。
- [ ] reconciler 目标筛选仍是 label key exists，范围过宽（internal/ctrl/reconciler/container_reconciler.go）。
- [ ] workload 更新未做冲突重试/退避/失败分类（internal/ctrl/reconciler/container_updated.go）。
- [ ] 删除逻辑未防止业务主容器被清空（internal/ctrl/reconciler/container_removed.go）。
- [ ] 注入链路指标缺失（当前 metrics 未覆盖注入路径，见 pkg/metrics/metric.go）。
- [ ] mutate handler 与端到端收敛测试缺失（现有测试主要是纯函数）。

## 4. 里程碑与交付

## 里程碑 M1（P0）：规则说明与主键修正

### 交付项
- [ ] 定义注入规则文档（一个源文件 + 一页说明）：
  - [ ] label key 表示“开启注入”。
  - [ ] label value 表示“目标容器模板名”。
  - [ ] 注入来源标记（annotation）统一。
  - [ ] 规则边界写清楚：当前只保证单 namespace 场景。
- [ ] 固化当前主键方案：在单 namespace 前提下继续使用 containerName 作为缓存主键。
- [ ] 在文档中补充扩展条件：一旦切到多 namespace，主键升级为 namespace/containerName。
- [ ] Admission 入参校验补齐（Kind/Operation/SubResource）。

### 验收标准
- [ ] 团队成员只看文档就能回答“什么时候注入、注入哪个模板、不注入时如何处理”。
- [ ] 非 CREATE Pod 请求不会进入注入逻辑。
- [ ] cache miss 行为与 failurePolicy 语义一致且文档明确。

### 最小测试
- [ ] webhook：label key/value 规则测试（覆盖注入和不注入）。
- [ ] webhook：Kind/Operation/SubResource 非法输入。
- [ ] webhook：cache miss 行为断言。
