# 注入模块优化指南（定制版）

## 1. 文档目的

这份指南用于你当前仓库的注入模块补全与优化，目标是两件事：

1. Pod 创建时自动注入容器。
2. 容器定义发生变更时，相关工作负载可以滚动收敛到新版本。

该指南基于当前项目代码结构与现有设计思路，强调“先补关键能力，再做结构优化”，避免把补全任务误做成大重构。

## 2. 输入材料与范围

### 2.1 输入材料

- docs/思路.md
- docs/architecture-and-request-flow.md

### 2.2 现状代码入口

- 启动与依赖装配：internal/ctrl/logic.go
- Admission 注入逻辑：internal/ctrl/webhook/aggregation.go
- 容器变更与工作负载更新：internal/ctrl/reconciler/container_reconciler.go

### 2.3 本次范围

- webhook 与 controller 的职责边界。
- 共享缓存的读写边界与一致性语义。
- Kubernetes Client 的依赖收敛。
- 注入契约、失败语义、可测试不变量。

### 2.4 非范围

- 数据面代理转发逻辑。
- 注入模块之外的业务功能。
- 一次性完成全仓库大重构。

## 3. 当前架构快照

### 3.1 当前事实

1. 启动层统一创建 cache，并注入到 webhook 和 controller。
2. webhook 读取注入标签与 cache，返回 JSON Patch。
3. controller 监听容器定义变更，更新 Deployment/StatefulSet。
4. kube client 在多个子模块中直接使用。

### 3.2 当前耦合类型

- 必要耦合：
  - 注入契约共享（标签语义、容器字段、patch 结构）。
- 可优化耦合：
  - cache 读写接口偏弱类型，语义不够清晰。
  - 业务模块持有过宽 kube client 能力。
  - reconcile 选择逻辑与实际更新逻辑混在一起。

## 4. 补全优先的方法（防止任务膨胀成重设计）

你已经有可运行的 controller，因此本节采用“能力补全优先”原则：

1. 不改业务语义：先保证当前行为不变。
2. 只补关键缺口：优先幂等、收敛、失败处理、可观测性。
3. 小步重构：仅在必要处抽接口，不追求一次性架构重排。

### 4.0 先做补全清单（而不是先改架构）

必须先补齐这些能力，再决定是否继续重构：

1. webhook 注入幂等保护。
2. controller 的冲突重试与失败分类处理。
3. Add/Update/Delete 收敛测试。
4. 最小监控指标（成功率、失败率、冲突次数、收敛耗时）。

产出：
- 一份补全验收清单（Done/Not Done）。

### 第一步：先写契约，不写实现

只定义三份契约：

1. Admission 输入与注入决策契约。
2. 注入模板契约。
3. 容器变更后的滚动更新契约。

产出：
- 1 页契约说明（字段、默认值、兼容策略）。

### 第二步：先定边界和所有权

必须明确：

1. 注入模板由谁作为单一真相。
2. 目标工作负载由谁筛选。
3. 共享状态谁可写，谁只读。
4. 谁可以执行 Kubernetes 更新操作。

产出：
- Allowed/Forbidden 依赖矩阵。

### 第三步：先定失败语义

每条链路都要定义失败行为：

1. webhook 读 cache 未命中。
2. Admission 解析失败。
3. workload 更新冲突。
4. 部分目标更新失败。

产出：
- 失败策略表（重试、退避、跳过、拒绝）。

### 第四步：先定不变量和验收

1. 幂等：同一事件重复处理不重复注入。
2. 安全：删除注入容器不能使 pod 容器列表为空。
3. 收敛：容器更新后，目标 workload 最终一致。
4. 隔离：webhook 不依赖 controller 的运行时调用链。

产出：
- 合同测试清单。

## 5. 目标架构方向

### 5.1 职责拆分

- webhook：
  - 解析 AdmissionReview。
  - 进行注入决策。
  - 生成稳定、可重放的 patch。
- controller：
  - 监听容器定义事件。
  - 维护注入模板投影视图。
  - 执行幂等的 workload 更新。
- projection store（共享读模型）：
  - 为 webhook 提供只读、类型明确的数据。
  - 单写者原则。
- 启动编排层：
  - 依赖装配、生命周期管理、配置注入。

### 5.2 推荐依赖规则

- 允许：
  - 启动层 -> 全部模块。
  - controller -> 读写端口 + 更新端口。
  - webhook -> 只读端口 + patch builder。
- 禁止：
  - webhook -> controller 具体实现。
  - webhook -> 直接执行 workload 更新。
  - 业务模块直接依赖全局构造函数形成隐式依赖。

### 5.3 Kubernetes 访问端口建议

- WorkloadReader：查询目标 Deployment/StatefulSet。
- WorkloadWriter：执行更新与冲突重试。
- EventWatcher：订阅并分发容器定义事件。

## 6. 优化待办（按优先级）

### P0（必须做）

1. 固化注入契约与标签语义。
2. 增加注入幂等保护，避免重复 sidecar。
3. controller 中拆分“目标选择”和“更新执行”。
4. 补齐 Add/Update/Delete 收敛测试。

### P1（高优先）

1. 引入强类型投影接口，替代弱类型读取。
2. 增加冲突重试策略与相关指标。
3. 增加滚动更新进度与失败可观测性。

### P2（增益项）

1. 增加 dry-run 规划模式。
2. 增加按 namespace/label 的灰度更新策略。
3. 视需要增加投影版本或过期策略。

## 7. 设计评审检查清单

1. 注入契约是否明确且可演进。
2. 每条写路径是否有唯一 owner。
3. 所有重试是否有边界且可观测。
4. 重复/乱序事件是否仍幂等。
5. 失败回滚条件是否定义清楚。
6. 是否覆盖成功路径与至少 3 条失败路径测试。

## 8. 最值得练手的模块（结论）

推荐模块：

容器变更滚动更新引擎（controller 侧）

原因：

1. 架构价值高：可以展示边界、依赖倒置、模块化能力。
2. 分布式语义强：能讲清幂等、最终一致性、冲突处理。
3. Kubernetes 实战强：事件、筛选、更新、收敛全覆盖。
4. 面试讲述完整：问题、方案、权衡、结果都可量化。

## 9. 这个“引擎”具体要做什么

### 9.1 目标

把“容器定义变化”可靠地转化为“相关 workload 模板更新并最终收敛”。

### 9.2 输入

1. 事件输入：容器定义 Add/Update/Delete。
2. 当前状态：目标 namespace、标签、现有 workload 模板。
3. 策略输入：幂等键、冲突重试、更新顺序。

### 9.3 输出

1. 变更计划：哪些 workload 需要更新、更新什么字段。
2. 执行结果：成功/失败清单、失败原因。
3. 观测信号：耗时、冲突次数、失败率、收敛状态。

### 9.4 内部流程

1. Plan：根据事件筛选目标 workload 并生成更新计划。
2. Apply：逐个执行更新，处理冲突与重试。
3. Verify：检查模板是否达到目标状态。
4. Report：记录指标与结构化日志。

### 9.5 最小可交付版本

1. 先支持 Add 事件。
2. 只覆盖 Deployment。
3. 具备幂等保护与冲突重试。
4. 提供最小指标与回归测试。

## 10. 7-10 天练手路线

### 第 1-2 天：设计定稿

1. 写契约文档（字段与兼容策略）。
2. 写依赖矩阵（允许/禁止）。
3. 写 Add 事件时序图。

### 第 3-5 天：最小实现

1. 抽离 rollout planner。
2. 增加 WorkloadReader/WorkloadWriter 端口。
3. 接入现有 controller 路径，保证行为不变。

### 第 6-7 天：测试与观测

1. 加幂等与收敛测试。
2. 加冲突与部分失败测试。
3. 加最小指标与日志字段。

### 第 8-10 天：面试化沉淀

1. 写 1 份 ADR（关键取舍）。
2. 准备讲述模板：问题 -> 设计 -> 权衡 -> 故障 -> 结果。

## 12. 下一步建议（补全导向）

1. 先做“补全版里程碑 1”：仅补 Add 事件幂等与冲突重试。
2. 再做“补全版里程碑 2”：补 Update/Delete 收敛与失败策略。
3. 每个里程碑都必须包含对应测试与指标，再进入下一步。
