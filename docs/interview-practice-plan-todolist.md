# 面试导向实践学习计划 TodoList

## 0. 计划原则（先做再讲）

- [ ] 每个主题都走完一轮：问题定义 -> 设计文档 -> 实现 -> 测试验证 -> 复盘文档
- [ ] 每个主题都产出可复用材料：1 份设计文档 + 1 份实现记录 + 1 份故障/边界复盘
- [ ] 不追求背诵，追求“自己亲手做过并能解释取舍”
- [ ] 所有文档都放在 docs 目录，命名统一，便于回顾

---

## 1. 文档与设计要求（保留并细化）

### 1.1 设计文档模板要求（每个主题必写）

- [ ] 背景与问题：当前痛点、触发场景、失败模式
- [ ] 目标与非目标：明确要解决什么、不解决什么
- [ ] 关键指标：正确性、延迟、吞吐、可用性、恢复时间
- [ ] 方案候选对比：至少 2 个方案，写清取舍理由
- [ ] 架构图与关键流程：请求路径、状态流转、异常分支
- [ ] 数据结构与状态模型：字段、并发模型、生命周期
- [ ] 一致性与幂等策略：重复事件、乱序、重放、并发覆盖
- [ ] 降级与容错：超时、重试、熔断、回退路径
- [ ] 可观测性：指标、日志、告警、排障入口
- [ ] 风险与回滚：上线风险、开关策略、回滚步骤
- [ ] 验收标准：功能、性能、稳定性、可运维性

### 1.2 实现记录文档要求（每个主题必写）

- [ ] 变更范围：涉及模块、接口、配置、部署清单
- [ ] 实现步骤：按提交顺序说明做了什么
- [ ] 与设计偏差：实际实现与设计差异及原因
- [ ] 测试结果：单测、并发测试、回归测试结论
- [ ] 已知问题：暂未解决项、影响范围、后续计划

### 1.3 复盘文档要求（每个主题必写）

- [ ] 本次踩坑清单：根因、触发条件、复现方式
- [ ] 误判与修正：最初判断错在哪里
- [ ] 关键权衡：为何不是另一种实现
- [ ] 下一步优化：按 P0/P1/P2 列优先级

## 3. 节奏建议（长期、低压力）

- [ ] 每 2 周完成 1 个主题闭环（设计 + 实现 + 复盘）
- [ ] 每周至少 1 次代码级回顾，更新文档中的“已知问题”
- [ ] 每个主题至少保留 1 个未完成优化项，作为后续面试延展点
- [ ] 每月做一次全局复盘：删除过时结论，补充新证据

## 4. 面试准备方式（不靠背诵）

- [ ] 每个主题准备 3 个你亲手解决的问题案例
- [ ] 每个案例固定回答结构：现象 -> 根因 -> 方案对比 -> 落地 -> 结果 -> 反思
- [ ] 每个案例至少有一个失败尝试，能讲清为何失败
- [ ] 每个案例有可量化指标或测试结论支持
